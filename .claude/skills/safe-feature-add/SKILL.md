# safe-feature-add — 安全な機能追加

My Chaos AI Suiteは「既存機能を絶対に壊さない」「1回の依頼につき原則1機能」が最重要ルール。新機能を追加するときは必ず次の順で進める（順番を飛ばさない）。

## 0. 作業前のGit状態確認（最初に必ず行う）

- `git status` / `git status --short -- chaos-ai-suite/`で、未コミットの変更（ケイオス師匠の作業中のものかもしれない）がないか確認する。あれば、それを勝手に破棄・上書きしない
- `git log --oneline -1`（または`git rev-parse HEAD`）で現在のコミット位置を確認し、「何か問題が起きたらここまで戻せる」という復元地点を把握しておく。この時点で新しいコミットを作る必要はない（そのコミット自体が復元地点になる）
- 未コミットの変更が残っている場合は、それが今回の作業に関係ないものなら先にケイオス師匠へ確認する（自分の変更と混ざらないようにするため）

## 1. 着手前の調査

- `chaos-ai-suite/README.md`と`.claude/rules/project-context.md`で、似た機能が既にないか・関連する既存コンポーネントがないか確認する
- 触る予定のファイル・型定義（`packages/shared`）・API（`packages/backend`）・保存先（localStorage/IndexedDBのキー）を洗い出す
- `.claude/rules/decisions.md`を確認し、過去の設計判断（データ保存方針・AI呼び出しパターン等）と矛盾しないか確認する
- 有料API・新規npmパッケージが必要なら、目的・無料代替の有無・料金発生の可能性・既存環境への影響を先にケイオス師匠へ説明し、実装前に確認を取る

## 2. 実装方針（変更範囲を最小化する）

- 新しいAI呼び出しは`llmClient.ts`の`callTool`（ツール強制呼び出し）パターンに合わせる
- データ保存はサーバーではなく、localStorage（小さいデータ）またはIndexedDB（画像・音声・動画等のBlob）
- 画面は既存パターン（サイドバーのカード → フルスクリーンモーダル`fixed inset-0 z-[70]`、縦1列、`office-*`トークン、共通クラス定数）を踏襲する（詳細: `.claude/rules/mobile-first.md`）
- 既存コード・既存デザイン・設定ファイル・環境変数を、明確な理由なく削除・全面置換しない
- 今回の機能に直接関係しないファイルは触らない。「ついでに直す」を避け、無関係な整形・リファクタを混ぜない

## 3. 実装中（1回につき1機能）

- 1回の依頼につき原則1機能。関係のない大規模リファクタや別機能の変更を混ぜない
- 保存データ構造を変える場合は、既存データとの互換性・移行方法を先に確認する

## 4. 実装後の確認（利用可能な範囲ですべて実行する）

`chaos-ai-suite/`直下で、`package.json`に実際に存在するコマンドだけを実行する（存在しないコマンドを推測で実行しない）。

- **TypeScript型チェック**: `npm run typecheck`（編集のたびにPostToolUseフックが自動実行する設定が入っているが、まとまった変更の後は改めて手動でも実行して確認する）
- **ビルド**: `npm run build`
- **利用可能なテスト**: `npm run test --workspace packages/backend`（フロントエンドには現時点で自動テストがないため、`npm run dev:frontend`で実際にブラウザ操作して確認する）
- **スマホ表示への影響**: `.claude/rules/mobile-first.md`の必須項目（横スクロールなし・文字/ボタン見切れなし・タップ領域確保・縦1列基本・モーダルが`fixed inset-0 z-[70]`パターンに沿っているか）をコードで確認する。可能であれば`mobile-ui-reviewer`サブエージェントでもレビューする
- 可能であれば`code-reviewer`・`security-reviewer`サブエージェントでもレビューする

## 5. 完了報告（必ず日本語で、次の4項目を含める）

1. **変更ファイル** — ファイルごとに何を・なぜ変更したか
2. **確認結果** — 型チェック・ビルド・テスト・スマホ表示確認のそれぞれについて、実行したものは結果を、実行できなかったものは「未確認」と明記する（やっていないことを「確認済み」と書かない）
3. **残存リスク** — 今回の変更で影響する可能性がある既存機能・未確認のまま残る項目・実機確認が必要な項目
4. **戻し方** — 問題があった場合にどう元に戻せるか（例: 「手順0で確認したコミット〈ハッシュ〉より前の状態に戻せます」「未コミットならファイルを元に戻すだけで解消します」等、具体的に）
