# bug-investigation — 原因調査を優先するバグ修正

「推測で実装しない」というプロジェクトの原則に沿い、コードを直す前に原因を確定させる。

## 1. 再現条件の確認

- どの画面・どの操作で・何が起きたか（期待した動きと実際の動き）をケイオス師匠の報告から整理する。不明な点は推測せず確認する
- 可能ならブラウザのコンソールエラー・ネットワークタブ・バックエンドのログを確認する

## 2. 原因調査（修正より先に行う）

- `Grep`/`Glob`で関連コードを特定する。フロントエンドの表示問題なら対象コンポーネント、データが消える問題ならlocalStorage/IndexedDBの保存・読み出し処理、AI応答がおかしい問題なら`llmClient.ts`の`callTool`呼び出しとプロンプト
- 関連する型定義（`packages/shared`）まで遡り、型と実際のデータ構造が一致しているか確認する
- 直近の変更が原因の可能性があれば`git log` / `git diff`で確認する
- 原因が複数考えられる場合は、コードから確認できる範囲で絞り込み、確認できない外部要因（APIの仕様変更等）は「確認できませんでした」とケイオス師匠に報告する

## 3. 修正方針

- 原因箇所への最小限の修正にとどめる。ついでの大規模リファクタをしない
- 同じ原因が他の画面・機能にも波及していないか確認する（共有型・共通ユーティリティの場合は特に）

## 4. 修正後の確認

- `npm run typecheck` / `npm run build`
- 関連する`npm run test --workspace packages/backend`
- 元の再現条件で直っているか確認する（可能ならPlaywright等での動作確認）
- 同じ操作の周辺で新しい不具合を生んでいないか確認する
- `.claude/rules/testing.md`の完了報告形式で、「何が原因で・何を直したか」を初心者向けの日本語で報告する
