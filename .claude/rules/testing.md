# テスト・品質確認ルール

## 実行可能なコマンド（package.jsonで確認済み。存在しないコマンドを推測で実行しない）

すべて `chaos-ai-suite/` ディレクトリ直下で実行する。

| コマンド | 内容 |
|---|---|
| `npm run build` | shared→backend→frontendの順に全ビルド |
| `npm run typecheck` | 全ワークスペースの型チェック |
| `npm run test --workspace packages/backend` | バックエンドのテスト（`node:test`、実APIキー不要） |
| `npm run dev:backend` | バックエンド開発サーバー（ポート4000） |
| `npm run dev:frontend` | フロントエンド開発サーバー（ポート5173） |

**注意**: フロントエンドには自動テスト（vitest等）が設定されていない。フロントエンドの動作確認は「ビルド成功」＋「実際にブラウザで操作」で行う。

## 変更後に確認すること

- 構文エラー・型エラー・ビルドエラーがないか（`npm run typecheck` / `npm run build`）
- 参照切れ・未使用コード・重複処理がないか
- 既存機能への影響（関連するテストが通るか、関連画面が壊れていないか）
- スマホ表示（`mobile-first.md`）
- 読み込み中の表示・エラー時の表示が日本語で分かりやすいか
- データ保存が正しく行われるか（localStorage / IndexedDBの場合はリロード後の復元も確認）
- API失敗時の挙動（クラッシュせず、日本語でエラー表示されるか）
- 秘密情報がコードやログに混入していないか
- コンソールエラーが出ていないか

## 自己レビュー（実装をそのまま完成扱いにしない）

実装後、第三者のレビュアーとして次を確認する。

- 依頼内容とズレていないか
- 既存機能を壊していないか
- 不要な変更をしていないか
- もっと単純な方法がなかったか
- スマホで使いにくくないか
- 同じ処理を重複していないか
- エラー処理が不足していないか
- セキュリティ上の問題がないか
- 料金が増える実装になっていないか
- ケイオス師匠が次に何をすればいいか分かるか

問題が見つかった場合は、可能な範囲で修正してから完成報告する。

## 完了報告の形式

1. **今回行ったこと** — 初心者向けの言葉で
2. **変更したファイル** — ファイル名と役割
3. **動作確認** — 実行したテスト・型チェック・ビルド
4. **既存機能への影響** — 影響なし、または影響する可能性がある箇所
5. **未確認事項** — 確認できなかった項目があれば
6. **ケイオス師匠が次にすること** — ユーザー操作が必要な場合のみ、番号付きで
