# 10 アプリ仕様

- **ステータス**: 構想段階(Phase 3 で本格化。データ設計だけ先行して固定)
- **最終更新**: 2026-07-16

将来のアプリ化に向けた仕様の置き場です。
現段階で最も重要なのは、**カードデータをアプリからそのまま読める形で管理しておくこと**です。

## コンセプト

- アプリを開く体験 = 「時のゆりかごの扉をくぐる」
- 忙しい日常の中で、1日数分の「心を整理する時間」を提供する
- 通知や演出で急かさない。開かない日があっても責めない

## コア機能(構想)

| 優先度 | 機能 | 概要 |
|---|---|---|
| P0 | 今日の一枚 | 1日1枚カードを引き、メッセージを読む |
| P0 | カード図鑑 | 78枚を眺められる。引いたカードの記録 |
| P1 | 心のメモ | カードと一緒に短い気持ちを書き留める |
| P2 | キャラクター案内 | ルミナ達がガイドとして登場する |
| P3 | コミュニティ連携 | 【未設定】(慎重に検討) |

## データ設計方針(現段階で確定)

1. カードデータの正は Card Database の Markdown(YAMLフロントマター+本文)
2. アプリ用には [card-schema.json](../card-database/schema/card-schema.json) 準拠の JSON へ変換して配信する
3. Card ID(例: `LA-M-01`)を全システム共通の不変キーとする(`LA-M-00` はテンプレート専用予約IDのため配信データに含めない)
4. 画像はファイル名に Card ID と版番号を含め、テキストとは独立に差し替え可能にする

```
Markdown (正) ──変換──▶ cards.json ──▶ アプリ / Web / API
```

## 技術方針【仮設定】

- 最初は Web アプリ(PWA)から始め、ストアアプリ化は需要を見て判断
- スマホのブラウザだけでも開発・運用状況を確認できる構成を優先
- 特定のフレームワークへの依存を最小化(データ層を独立させる)
