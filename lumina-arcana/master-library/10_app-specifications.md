# Phase1 UX Master Specification(Version 1.0・原文)

## ■ プロジェクトコンセプト

Lumina Arcana は「オラクルカードアプリ」ではありません。

世界中の人が安心して帰ってこられる場所『時のゆりかご』を体験として提供することを目的とします。

最優先する価値

- 安心
- 温もり
- 余白
- 静寂
- 自然
- 生命
- 対話
- 自己受容

派手なゲーム演出は禁止。

世界そのものがユーザーを迎え入れることを最優先してください。

## ■ 起動フロー

アプリ起動
→ オープニングムービー(スキップ可能)
→ トップ画面
→ Card Ceremony
→ 画面① カードとの出会い
→ 画面② カードを読み解く
→ ジャーナル
→ 水の波紋
→ トップ画面へ戻る

## ■ オープニングムービー

**目的**: 世界へ迎え入れる儀式。

**内容**:

- 植物の枝葉が絡み合い「時のゆりかご」が完成する
- 朝→昼→夕→夜へ自然に時間が流れる
- 最後に流れ星
- 揺り籠に光が灯る
- ロゴ表示
- おかえりなさい

動画はスキップ可能。
BGMはトップ画面へ自然に引き継ぐ。

## ■ トップ画面

UI Master v1.0 採用

背景は世界そのもの。ユーザーを急がせない。

画面内では

- 葉が揺れる
- 蛍
- 光の粒
- 月明かり
- 水面

など最低限の環境アニメーションのみ。

UIは固定。背景のみ季節変更可能。

## ■ Card Ceremony

カードは「引く」ではない。**「カードが訪れる」**体験として設計する。

演出:

① 今日のカード押下
→ ② ボタン沈み込み
→ ③ 水の波紋
→ ④ 揺り籠発光
→ ⑤ 光の粒が集まる
→ ⑥ カード生成
→ ⑦ カード浮遊
→ ⑧ ユーザータップ
→ ⑨ カード反転
→ ⑩ カード表示

ガチャ演出は禁止。静かな儀式を目指す。

## ■ 画面① カードとの出会い

**目的**: カードを見る時間。カードを感じる時間。カードが主役。

**表示**:

- カード(大きく表示)
- 今日のあなたへ
- 短いメッセージ
- 「メッセージを読み解く」

情報量は最小。余白を最大。カードを眺める時間を作る。

## ■ 画面② カードを読み解く

**表示**:

今日のあなたへ

深く読み解く

- キーワード
- 意味
- アドバイス
- 注意点

→ ジャーナル
→ 過去を見る

画面①より情報量は多い。読みやすさを優先。

## ■ ジャーナル

**目的**: 心を整理する場所。

**入力**:

- 今日感じたこと
- 気付き
- メモ

**機能**:

- 保存
- 履歴
- 検索(将来)

## ■ 画面遷移

世界観を壊さない。

画面切替は

- 波紋
- フェード
- 光
- 風

のみ。派手なトランジションは禁止。

## ■ 管理ページ

ロック番号: 管理画面から変更可能。実際の番号は仕様書には記載せず、環境設定(環境変数等)で管理する(ユーザー指示 2026-07-17)

管理画面から変更可能:

- 背景画像
- 動画
- ロゴ
- カード画像
- カード文章
- カテゴリ
- BGM
- ボタン画像
- アイコン
- トップ画像
- リアル画像
- アニメ画像
- イベント画像
- 季節背景

スマホ対応・PC対応。
画像アップロードで即反映できる構造。

## ■ 世界設定(将来)

管理画面から

- 春 / 夏 / 秋 / 冬
- 雨 / 雪
- 満月 / 新月
- イベント

を変更できる。

変更すると

- 背景
- BGM
- 波紋色
- 光色
- トップ画面

が一括変更される。

## ■ カード画像(将来)

カードは Touch または Double Tap で切替可能。

アニメ版 → リアル版 → アニメ版(クロスフェード)。

カード情報は共通。

## ■ デザイン思想

引き算の美学。

情報を詰め込まない。余白を作る。安心を作る。カードを主役にする。

世界が呼吸しているように感じること。

## ■ Lumina Arcana ブランドルール

Lumina Arcana は

占いアプリではない。ゲームでもない。SNSでもない。AIチャットでもない。

ここは『時のゆりかご』。

世界中の人が安心して帰ってこられる場所。

カードは未来を当てるものではなく、今日という一日に寄り添う存在。

世界そのものが「おかえり。」と迎えてくれる体験を提供する。

これが Lumina Arcana のブランド哲学である。

# データ設計方針(Claude Code 管理・確定済み)

1. カードデータの正は Card Database の Markdown(YAMLフロントマター+本文)
2. アプリ用には [card-schema.json](../card-database/schema/card-schema.json) 準拠の JSON へ変換して配信する
3. Card ID(例: `LA-M-01`)を全システム共通の不変キーとする(`LA-M-00` はテンプレート専用予約IDのため配信データに含めない)
4. 画像はファイル名に Card ID と版番号を含め、テキストとは独立に差し替え可能にする
5. カード画像はアニメ版/リアル版の2系統を同一 Card ID に紐づけられる構造とする(UX仕様「カード画像(将来)」対応)

```
Markdown (正) ──変換──▶ cards.json ──▶ アプリ / Web / API
```

## UX仕様とデータ項目の対応

| UX仕様の表示 | Card Database の項目 |
|---|---|
| 画面① 短いメッセージ | 基本メッセージ(messages.base) |
| 画面② キーワード | keywords |
| 画面② 意味 | ストーリー+基本メッセージ(story / messages.base) |
| 画面② アドバイス | 今日の行動(messages.action)+各運勢 |
| 画面② 注意点 | 注意点(messages.caution) |

# 技術方針【仮設定】

- 最初は Web アプリ(PWA)から始め、ストアアプリ化は需要を見て判断
- スマホのブラウザだけでも開発・運用状況を確認できる構成を優先
- 特定のフレームワークへの依存を最小化(データ層を独立させる)
