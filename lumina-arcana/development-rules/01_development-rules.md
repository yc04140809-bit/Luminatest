# 01 開発ルール(基本原則)

- **最終更新**: 2026-07-16

本ファイルは [00_project-rules.md](./00_project-rules.md)(Project Rules v1.0)と [06_ai-role-definition.md](./06_ai-role-definition.md)(AI Role Definition v1.0)——いずれもユーザー制定の最上位ルール——を実務に落とし込んだ運用規範です。矛盾する場合は 00・06 が優先されます。

**役割の要点**: 最終決定権はケイオス師匠(プロジェクトオーナー)。世界観・キャラ・カード内容は Lumina(ブランド統括AI)の担当。Claude Code は開発責任AI(構成・管理・設計・実装)であり、役割外の判断は行わず担当者へ確認する。
判断に迷った場合の優先順位: ①ブランド理念 → ②世界観 → ③キャラクター設定 → ④カード設定 → ⑤実装

## 基本原則

1. **理念が最上位**: すべての作業はブランド理念「安心・癒し・優しさ・希望・心を整理する時間・依存させない・恐怖で煽らない・未来を断定しない・自分で答えを見つけられる設計」([00_project-rules](./00_project-rules.md) / [master-library/00章](../master-library/00_brand-philosophy.md))に従う。ブランド理念より利便性を優先しない
2. **Single Source of Truth**: 設定の正は `master-library/`。同じ情報を複数の場所に書かず、リンクで参照する
3. **土台優先**: 実装(コード)より構造とドキュメントの整合性を優先する。5年後・10年後に読める状態を保つ
4. **スマホで運営可能**: すべてのデータはプレーンな Markdown / JSON で管理し、スマホのGitHubアプリ・ブラウザだけでも読み書きできる形式を守る(特殊なツールが必須になる構成にしない)
5. **小さく確定、大きく拡張**: 未確定事項は管理タグ(下記)で明示して枠だけ作る。推測で補完しない

## 管理タグ(状態管理・提案の表記)

正式定義は [00_project-rules.md](./00_project-rules.md) の「タグ運用ルール」(v1.1・ユーザー制定)。

| タグ | 意味 | 状態を変更できる人 |
|---|---|---|
| 【未設定】 | まだ内容が存在しない | ユーザー |
| 【仮設定】 | 検討中・サンプル・ドラフト・テンプレート例 | ユーザー |
| 【要確認】 | ユーザー確認待ち | ユーザー |
| 【正式設定】 | Master Libraryで正式承認済み | ユーザー |
| 【廃止】 | 過去設定。履歴として保存するが現在は使用しない | ユーザー |
| 【提案】 | AIからの改善案。決定事項ではない。採用・不採用はユーザーが判断 | ユーザー |

- タグは必ず全角隅付き括弧【】で書き、上記以外の類似表記(【仮】等)は使用しない
- AIはタグ付きの項目を、ユーザーの指示なしに確定・削除しない
- YAMLフロントマターの `status`(planned/draft/review/released/revising)との対応目安: planned≒【未設定】、draft≒【仮設定】、review≒【要確認】、released≒【正式設定】

## AIエージェントの行動規範

このプロジェクトでは複数のAIが作業することを前提とします。

1. **作業前に読む**: `README.md` → `development-rules/` 全部 → 作業対象ディレクトリの `README.md`
2. **勝手に設定を創作しない**: 世界観・キャラクター・設定・ストーリー・カード内容を創作しない。【未設定】【要確認】【仮設定】の項目は、ユーザーの指示があるまで確定させない。判断に迷う場合は必ず質問する
3. **改善案は【提案】として提示**: 決定事項として反映しない。採用・不採用は必ずユーザーが判断する
4. **テンプレートに従う**: 項目の追加・削除・並べ替えをしたい場合は、個別ファイルではなくテンプレート自体の改訂として【提案】する
5. **ID・正式表記を厳守**: Card ID / CHR-xxx / 用語集の表記を変更・省略しない
6. **破壊的変更は確認を取る**: ファイル削除・ID変更・用語廃止・キャラ設定の大変更は、実行前にユーザーの承認を得る
7. **変更を記録する**: [05_更新ルール](./05_update-rules.md) に従い「変更履歴・変更理由・影響範囲」を残す

## 文章を書くとき

- カード・キャラのセリフ・SNS文面など、利用者の目に触れる文章はすべて [08_ライティングガイドライン](../master-library/08_writing-guidelines.md) に従う
- ドキュメント(この種の管理文書)は簡潔・明確を優先してよい

## Git 運用

- コミットメッセージは日本語可。「何を・なぜ」を1行目に書く
- 1コミット = 1つのまとまった変更(カード1枚の追加、章1つの改訂など)
- main への直接コミットは避け、作業ブランチ経由を基本とする
