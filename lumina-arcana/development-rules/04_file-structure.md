# 04 ファイル構成(内部構造)

- **最終更新**: 2026-07-16

ファイルの「中身の書き方」のルールです。人間にもAIにもプログラムにも読みやすい形を保ちます。

## 共通ルール

- 形式は Markdown(`.md`)。データスキーマのみ JSON
- 文字コードは UTF-8、改行は LF
- 見出しは `#` から始め、レベルを飛ばさない(`#` → `##` → `###`)
- 表で管理できるものは表にする(スマホでの視認性と機械処理のため)
- 未確定の内容は管理タグ【未設定】【要確認】【仮設定】を必ず付ける。AIの改善案は【提案】を付ける(定義は [01_開発ルール](./01_development-rules.md) の「管理タグ」参照)

## データファイル(カード・キャラ・世界エントリ)

**YAMLフロントマター + 本文** の二層構造。

```markdown
---
card_id: "LA-M-00"     # ← 機械可読データ(アプリ・スクリプトが読む)
title_ja: "..."
status: "draft"
---

# 見出し               ← 人間可読の本文(長文・物語・メッセージ)
...
```

- **フロントマター**: ID・タイトル・属性・ステータスなど、一覧化・検索・変換に使う短いデータ
- **本文**: ストーリー・メッセージなどの長文
- フロントマターのキー名はテンプレートの定義から変更しない(変換スクリプトが壊れるため)

## ドキュメントファイル(Master Library・ルール類)

冒頭に以下のメタ情報を置く。

```markdown
# 章タイトル

- **ステータス**: ドラフト / 正式版 vX.X / 運用中
- **最終更新**: YYYY-MM-DD
```

末尾に「更新履歴」の表を置く。

## ステータスの共通語彙

| 値 | 意味 |
|---|---|
| `planned` | 枠のみ・未着手 |
| `draft` | 作成中 |
| `review` | レビュー待ち |
| `released` | 正式版 |
| `revising` | 公開後の改訂中 |

## リンクのルール

- リポジトリ内のリンクは**相対パス**で書く(GitHub上でもローカルでも辿れるように)
- 外部サービスへのリンクには取得日をメモする(リンク切れ対策)
