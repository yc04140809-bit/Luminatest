## 第1章:Claude Codeとは〜「AIに書かせる」から「AIに任せる」へ〜

### 一言でいうと「自分で手を動かすAI」

ChatGPTやClaudeの普通のチャットは、こうです。

> あなた「コード書いて」
> AI「はい、コードです(貼り付けはあなたがやってね)」

Claude Codeは、こうなります。

> あなた「この機能作って」
> AI「はい。ファイル読みました。ここを直します。書き換えました。動作確認しました。GitHubに保存(コミット)しました」

**違いは「実行権限」です。**

普通のチャットAIが「アドバイスをくれる先生」だとしたら、Claude Codeは「実際に作業してくれる同僚」。

#### 図解イメージ案

```
【普通のAIチャット】
 あなた ⇄ AI
 (コードのやり取りは全部手動。コピペ地獄)

【Claude Code】
 あなた → 指示 → AI
                 ↓
        ファイルを読む・書く・直す・保存する
                 ↓
 あなた ← 「終わりました」報告
```

介護の現場でたとえるなら、

- 普通のAI=「記録の書き方を教えてくれる先輩」
- Claude Code=「記録を代わりに書いて、確認まで回してくれる後輩」

です。どっちが現場を回せるかは、言うまでもないですよね。

### スマホでも使えるの?→使えます(私が証明です)

Claude Codeは元々パソコンのターミナル(黒い画面)で動くツールでしたが、今はWeb版・スマホアプリからも使えます。

私の開発環境は、

- スマホ(Android)
- GitHub(コードの保管庫。無料)
- Claude Code(Web/アプリから)

これだけ。パソコンを開いたのは、この半年で数回です。

## 第2章:Fable 5とは〜「Opusの上」が来た〜

### Claude 5ファミリーの第1号

Fable 5は、Anthropicの新世代「Claude 5ファミリー」の最初のモデルです。

ポイントは3つ。

| 項目 | 内容 |
|------|------|
| 位置づけ | 従来最上位だったOpusの**さらに上**の新ティア「Mythosクラス」 |
| 提供形態 | Fable 5は一般提供版。兄弟モデルのMythos 5は承認組織のみ |
| 得意分野 | 長い作業の一貫性、曖昧な指示の補完、コードの自律的な完遂 |

今までは「Opus=一番賢い」でした。
その上が、普通に使えるようになった。これが話題の理由です。

### 過去モデルとの違い:私の体感

スペック表は公式を見てください。私が書けるのは**体感**です。

**① 「途中で忘れない」**

以前のモデルだと、長い作業の後半で「あれ、最初に言った条件どこいった?」が時々ありました。Fable 5では、1,400行のHTMLを触らせても、冒頭で伝えた「介護現場の言葉づかいで」という条件が最後まで生きていました。

**② 「聞き返す前に確かめる」**

前は「〜という理解で合ってますか?」と聞き返されることが多かった。Fable 5は、先にファイルを読んで、リポジトリの過去の変更まで見て、「たぶんこういう意図ですね」と当ててきます。しかも大体当たってる。

**③ 「失敗の報告が正直」**

地味に一番大事です。前のモデルは「できました!」と言って、できてないことがありました。Fable 5は「テストは通りましたが、この部分は確認できていません」と言ってくる。**AIの報告を信じられるかどうか**は、スマホ運用だと死活問題なんです(画面が小さくて自分での確認コストが高いので)。

## 第3章:実際に本気で使った感想〜「ケアちゃん」が1日でできた日〜

### 事例①:介護研修資料AI「ケアちゃん」

私の本業は介護です。現場で一番面倒な仕事のひとつが**研修資料づくり**。

「身体拘束適正化研修の資料、来週までにお願い」

……これ、介護職なら胃が痛くなるやつです。

そこでClaude Code(Fable 5)にこう頼みました。

> 「介護施設の法定研修の資料を自動生成するWebアプリを作って。テーマを選んだら、スライド形式の研修資料と確認テストが出るように。難しい専門用語は現場の言葉に言い換えて」

結果、**1,424行のHTMLアプリが、その日のうちに動きました。**

驚いたのはここからです。私は「キャラクターをつけて」としか言っていないのに、

- 研修講師キャラ「ケアちゃん」の口調が全ページで統一されていた
- 既にサイトにあったアバター画像を勝手に(いい意味で)見つけて流用してくれた
- 「アプリアイコンにも設定しておきました」と報告してきた

**「アイコン設定して」なんて一言も言ってません。**

でも、スマホのホーム画面に置いて使う私の運用を、リポジトリの構成から推測したんだと思います。この「一歩先」が、Fable 5の一番の進化だと感じました。

### 事例②:実際のやり取り(ほぼ原文)

私のトップページ改修時の指示です。ひどい指示なので晒します。

> 私「トップをケイオスちゃんのチャット画面にして。バトルのやつはもういらない」

これだけ。普通なら「バトルのやつとは?」「チャット画面の仕様は?」と質問攻めのはず。

Fable 5の返答(要約):

> 「index.htmlのバトルモック部分を確認しました。既存のケイオスちゃんのキャラ設定(過去のコミットにあった仕様書)を踏襲して、チャット画面に置き換えます。背景は夜の部屋と昼の部屋を時間帯で切り替える形にしました」

**「時間帯で背景切り替え」は頼んでません。** でも、素材フォルダに bg_night_room と bg_pink_room の2枚があるのを見て、「2枚あるってことは切り替えたいんでしょ」と汲んだわけです。

正直、人間の後輩より察しがいい。

### 事例③:1,868行の書き換えを、スマホで寝転びながら

トップページの全面改修は、差分でいうと約1,900行の変更でした。

以前の私なら、これをチャットAIとやるには「コード貼って→修正案もらって→コピペして→エラー→また貼って」を数十往復。スマホだと現実的に無理でした。

Fable 5では、指示は3回。所要時間は待ち時間込みで1時間弱。私がやったことは、布団の中で指示を出して、できあがった画面をスマホで確認しただけです。

**「開発のハードルが下がる」ってこういうことか**、と実感しました。

## 第4章:ここは正直イマイチ〜金を払ってるから言わせてもらう〜

べた褒めだけの記事は信用しないでください。ここからが本音です。

### ① 賢すぎて「先回りが余計」なことがある

前章の「察しの良さ」、裏返すと**頼んでないことをやる**ということです。

私が「ボタンの色だけ変えて」と言ったとき、「ついでに周辺のデザインも統一しておきました」と、触ってほしくない箇所まで整えられたことがありました。良かれと思って、なんですけどね。

**対策:「〇〇だけ。他は触らないで」と明示する。** これで9割防げます(特典プロンプト④参照)。

### ② スマホでの差分(diff)確認がつらい

これはFable 5というよりスマホ運用の限界ですが、1,000行超の変更をスマホの画面で確認するのは、正直しんどい。横スクロール地獄です。

私は諦めて「変更点を介護記録みたいに要約して」と頼むようにしました。これが意外と正解で、**AIの報告が正直になったFable 5だからこそ成立する運用**です。前のモデルなら要約を信じ切れなかった。

### ③ 画像を扱うと途端に重い

サイト用の背景画像(1枚2MB超)をコミットさせたとき、pushにやたら時間がかかり、スマホの回線だと一度失敗しました。画像はAIに任せず、あらかじめ軽くしてから渡すのが吉です。

### ④ 失敗談:曖昧な指示で「大事なページ」が消えかけた

一番冷えた話をします。

「古いファイル、整理しておいて」と雑に頼んだら、私が手動で作った下書きHTMLを「未使用ファイル」と判断して削除候補に挙げられました。実行前に「これは削除して良いですか?一覧を確認してください」と確認してくれたので事なきを得ましたが、**ノールックでOKしてたら消えてました。**

教訓:**AIの確認には、面倒でも目を通す。** Fable 5は確認を求めるのが上手になった分、人間側が「はいはいOK」を押す機械になりがちです。ここは本当に注意。

### ⑤ 料金は「無料で無限」ではない

利用にはプラン契約が必要で、使い放題ではありません。私は月の後半、使用量を気にしながら使う日があります。副業の初期投資として見れば安いですが、「完全無料でイケる」と思って来ると裏切られるので先に書いておきます。

## 第5章:私はこう使っている〜介護職の1週間、AI開発の実録〜

私の実際の運用を晒します。全部スマホです。

### My Chaos AI SITE(自作サイト開発)

- キャラクター「ケイオスちゃん」のチャット画面をClaude Codeで実装
- 衣装差分・背景差分の管理もリポジトリごと任せる
- 「仕様書もコミットしておいて」と頼むと、次回以降その仕様書を勝手に読んで一貫性を保ってくれる ←これ最強

### AI社員プロジェクト

「AIを社員として雇う」がコンセプト。研修資料担当の「ケアちゃん」が第1号社員です。役割・口調・禁止事項を仕様書化してリポジトリに置く。すると、Claude Codeが毎回それを読んでから作業する。**人間の新人教育と全く同じ構造**なのが面白いところ。

### Threads分析

伸びた投稿・沈んだ投稿を貼って、「共通点を介護職に説明するつもりで言語化して」と依頼。出てきた仮説を翌週の投稿で検証。この記事の宣伝計画も、この方法で作っています。

### note制作

構成案→本文→タイトル案→サムネ案まで一気通貫。ただし**「体験」の部分だけは絶対に自分で書く**。そこをAIに書かせた記事は、読めば分かります。あなたが今読んでいるこの失敗談も、私の実話です。

### 画像生成プロンプト改善

生成AI用のプロンプトをClaude Codeに管理させ、「前回より背景がうるさかった。直して」と履歴込みで改善。プロンプトがバージョン管理されるので、**「あの時の神プロンプト」が消えない**。

### 開発記録

作業のたびに「今日やったことを日記形式でmemoに追記して」と一言。積み重なった記録が、そのままnoteのネタ帳になります。この記事の第3章は、ほぼその記録から書きました。

## 第6章:ChatGPTとの違い〜「どっちが上」ではなく「どっちが担当」か〜

毎日両方使っている私の結論:**比べるものじゃなく、組ませるもの**です。

### 比較表

| 項目 | ChatGPT | Claude Code (Fable 5) |
|------|---------|----------------------|
| 得意 | 会話、壁打ち、調べもの、雑談 | ファイル操作、コード、長い作業の完遂 |
| 例えると | なんでも相談できる友人 | 黙々と仕上げてくる職人 |
| アイデア出し | ◎ テンポが良い | ○ 硬めだが深い |
| 実装・修正 | △ コピペ地獄 | ◎ 全部やってくれる |
| 長い作業の一貫性 | △ 途中で迷子になりがち | ◎ 最後まで覚えている |
| スマホ運用 | ◎ アプリが軽快 | ○ 可能(diff確認は工夫が要る) |
| 向いてる人 | まず話したい人 | まず作りたい人 |

### 私の使い分け(実例)

1. **朝の通勤:ChatGPT**で「今日ケアちゃんに追加する機能、どれがいい?」と壁打ち
2. **休憩中:Claude Code**に決まった仕様を投げて実装させる
3. **帰宅後:**できたものをスマホで確認、ChatGPTに「この機能の紹介文を書いて」

つまり、**企画会議はChatGPT、実働はClaude Code**。
どちらか片方を解約しろと言われたら3日悩みます。それくらい役割が違う。

## 第7章:結論〜おすすめする人、しない人〜

### おすすめする人

- **何か作りたいものがある人**(サイト、ツール、ボット、なんでも)
- ChatGPTで「コードをコピペするのに疲れた」人
- スマホしかないけど開発してみたい人(←私です)
- AI副業の「作る側」に回りたい人
- 本業の面倒な作業(資料作成など)を自動化したい人

### おすすめしない人

- 調べもの・雑談が目的の人(ChatGPTで十分です)
- 「完全無料」を求める人
- AIの出力を一切確認せずに使いたい人(第4章④を読み返してください)

### 今日から始める3ステップ(スマホだけでOK)

「試してみよう」と思った今が、一番動けるときです。私が通った最短ルートを置いておきます。

1. **GitHubのアカウントを作る(無料・5分)**
   コードの保管庫です。スマホのブラウザで作れます。最初は空っぽでOK。
2. **Claudeのアカウントでプランを確認する**
   Claude Codeが使えるプランを公式サイトで確認して登録。ここだけお金の話なので、まず最小プランで様子見が私のおすすめ。
3. **最初の指示は「作りたいものの相談」から**
   いきなり「作って」ではなく、特典プロンプト①(後述)でAIに自分の状況を覚えさせてから。ここで差がつきます。

最初の成果物は「自己紹介ページ1枚」くらい小さくしてください。私も最初はそこからでした。**小さく成功する→楽しくなる→続く**。介護の自立支援と同じ理屈です。

### 今後どう進化すると思うか

Fable 5を使って確信したのは、進化の方向が「賢さ」から**「任せられる度」**に変わったことです。

次に来るのは、「私が寝ている間に、Threadsの反応を見てサイトを直しておいてくれるAI」だと思っています。実際、定期実行の仕組みはもうあります。**「AI社員」が比喩じゃなくなる日**は、たぶん想像より近い。

### 今後も検証していく理由

私は介護職です。エンジニアの「すごい活用術」は、正直まぶしすぎて参考にできませんでした。

だから私は、**「普通の人がスマホでどこまでやれるか」の記録係**でいようと思っています。失敗も含めて全部出します。それが一番、これから始めるあなたの役に立つはずだから。

次回は「AI社員ケアちゃんの育成記録」を予定しています。フォローして待っていてください。

### ① 初回セットアップ:AIに「現場」を覚えさせる

```
このリポジトリの全体像を把握して、CLAUDE.mdというファイルに
以下をまとめて保存してください。

・このプロジェクトの目的:【例:介護研修資料の自動生成アプリ】
・想定ユーザー:【例:介護施設の職員。ITは得意でない】

```

**ポイント:** これを最初に1回やるだけで、以降の指示が半分の長さで通ります。新人教育と同じです。

### ② 機能追加:「先回り」を味方につける

```
【追加したい機能】を実装してください。

進め方:
1. まず関連ファイルを読んで、実装方針を3行で私に報告
2. 私が「OK」と言ってから実装開始
3. 完了後、変更点を専門用語なしで箇条書きにして報告

私はスマホで確認するので、報告は短く、確認すべき画面を具体的に指定してください。
```

### ③ スマホ用・変更内容の要約(diff確認の代わり)

```
今回の変更内容を、ITに詳しくない上司に口頭で説明するつもりで
要約してください。

・何が変わったか(3行以内)
・ユーザーから見て変わる点
・壊れる可能性がある箇所と、その確認方法(スマホでできる方法で)
```

**ポイント:** 第4章②の対策そのものです。1,000行の差分を読む代わりに、これ。

### ④ 「余計なこと」防止の呪文

```
【直したい箇所】だけを修正してください。

制約:
・指定した箇所以外のファイル・デザイン・文言は一切変更しない
・「ついでに」の改善提案があれば、実行せずに提案だけ箇条書きで出す
・変更前に、触るファイルの一覧を先に見せてください
```

**ポイント:** 提案「だけ」させるのがコツ。良い提案は次回採用すればいい。

### ⑤ 開発日記の自動化(note のネタ帳になる)

```
今日の作業内容を dev-log.md に追記してください。

書式:
・日付
・やったこと(体験)
・つまずいた点と解決方法(気付き)
・次回やること(学び)

私がnoteの記事にそのまま使えるよう、
感情が動いたポイント(驚いた・困った・嬉しかった)があれば
それも記録してください。
```
